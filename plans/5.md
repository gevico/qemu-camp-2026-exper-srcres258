# Plan: Implement `vdot` Instruction Translation for Xg233ai

## Summary

Implement the `vdot` (INT32[16] vector dot product) instruction in QEMU's RISC-V TCG frontend. The instruction reads two 16-element INT32 vectors from memory, computes their dot product with INT64 accumulation, and writes the truncated result to a destination GPR.

## Current State

The Xg233ai extension already has 4 instructions implemented (`dma`, `sort`, `crush`, `expand`). The extension infrastructure (decode tree, translation hook, helper declarations, CPU config field, ISA registration, build system) is fully in place. The test file `tests/gevico/tcg/riscv64/test-insn-vdot.c` already exists and is registered in the build system (`Makefile.softmmu-target` line 32).

## What Needs to Change

### 1. Decode Tree: `target/riscv/xg233ai.decode`

Add the `vdot` instruction entry. The encoding is:

- funct7 = `1000110` (decimal 70)
- funct3 = `110`
- opcode = `1111011` (0x7B, custom-3)

```
vdot     1000110  ..... ..... 110 ..... 1111011  @r
```

This follows the same pattern as existing entries (e.g., `expand   0110110  ..... ..... 110 ..... 1111011  @r`).

### 2. Helper Declaration: `target/riscv/helper.h`

Add a `DEF_HELPER_FLAGS_3` declaration at line ~1292 (after the existing Xg233ai helpers):

```c
DEF_HELPER_FLAGS_3(vdot, TCG_CALL_NO_WG, tl, env, tl, tl)
```

Note the difference from existing helpers:
- Existing 4 helpers use `DEF_HELPER_FLAGS_4` with return type `void` — they only do memory operations.
- `vdot` uses `DEF_HELPER_FLAGS_3` with return type `tl` — it returns the dot product result to be written to `gpr[rd]`.

Parameters: `(env, rs1_addr, rs2_addr)` → returns `tl` (the dot product result).

### 3. Helper Implementation: `target/riscv/op_helper.c`

Add the `HELPER(vdot)` function after the existing `HELPER(expand)` (after line ~840):

```c
target_ulong HELPER(vdot)(CPURISCVState *env,
                          target_ulong rs1, target_ulong rs2)
{
    int64_t acc = 0;
    for (int i = 0; i < 16; i++) {
        int32_t a = (int32_t)cpu_ldl_data(env, rs1 + i * 4);
        int32_t b = (int32_t)cpu_ldl_data(env, rs2 + i * 4);
        acc += (int64_t)a * (int64_t)b;
    }
    return (target_ulong)acc;
}
```

Key points:
- Fixed 16-element loop per spec.
- Uses `cpu_ldl_data()` to load 32-bit values (matches existing helpers like `sort`).
- INT64 accumulator prevents intermediate overflow.
- Truncation to `target_ulong` (64-bit on RV64) is implicit via the cast.

### 4. Translation Function: `target/riscv/insn_trans/trans_xg233ai.c.inc`

Add `trans_vdot()` after the existing translation functions (after line ~55):

```c
static bool trans_vdot(DisasContext *ctx, arg_r *a)
{
    if (!ctx->cfg_ptr->ext_xg233ai) {
        return false;
    }

    TCGv rs1 = get_gpr(ctx, a->rs1, EXT_NONE);
    TCGv rs2 = get_gpr(ctx, a->rs2, EXT_NONE);
    TCGv rd = dest_gpr(ctx, a->rd);

    gen_helper_vdot(rd, tcg_env, rs1, rs2);
    gen_set_gpr(ctx, a->rd, rd);
    return true;
}
```

Key points:
- Uses `dest_gpr()` (not `get_gpr()`) for the destination register — this is the correct pattern for GPR-write instructions in QEMU.
- `gen_helper_vdot(rd, tcg_env, rs1, rs2)` — first arg is the return destination TCGv.
- `gen_set_gpr()` writes the TCGv result back to the architectural register.
- Existing helpers (dma/sort/crush/expand) don't write back to GPR, so they use `get_gpr` for all operands and `void` return. `vdot` differs because it writes a scalar result to `gpr[rd]`.
