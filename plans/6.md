# Plan: Implement `vrelu` Instruction Translation for Xg233ai

## Summary

Implement the `vrelu` (INT32 vector ReLU activation) instruction in QEMU's RISC-V TCG frontend. The instruction reads an INT32 array from source memory, applies `max(0, x)` element-wise, and writes results to destination memory. It supports in-place operation (source == destination).

## Current State

The Xg233ai extension has 5 instructions already implemented (`dma`, `sort`, `crush`, `expand`, `vdot`). All extension infrastructure is in place. The test file `tests/gevico/tcg/riscv64/test-insn-vrelu.c` already exists and is registered in the build system.

## Instruction Encoding

- funct7 = `1010110` (decimal 86)
- funct3 = `110`
- opcode = `1111011` (0x7B, custom-3)
- Format: `.insn r 0x7b, 6, 86, rd, rs1, rs2`

| Operand | Register | Meaning |
|---------|----------|---------|
| rd      | x[rd]    | Destination array base address |
| rs1     | x[rs1]   | Source array base address |
| rs2     | x[rs2]   | Number of elements N |

The instruction does **not** write back to any GPR — it only performs memory-to-memory operations.

### 2. Helper Declaration: `target/riscv/helper.h`

Add after line 1293 (after the existing Xg233ai helpers). Since `vrelu` is a memory-only operation with no GPR return, it uses `DEF_HELPER_FLAGS_4` with `void` return — same pattern as `dma`, `sort`, `crush`, `expand`:

```c
DEF_HELPER_FLAGS_4(vrelu, TCG_CALL_NO_WG, void, env, tl, tl, tl)
```

### 3. Helper Implementation: `target/riscv/op_helper.c`

Add after the existing `HELPER(vdot)` (after line ~852). Implementation follows the pseudocode from the datasheet:

```c
void HELPER(vrelu)(CPURISCVState *env, target_ulong rd,
                   target_ulong rs1, target_ulong rs2)
{
    int n = (int)rs2;

    for (int i = 0; i < n; i++) {
        int32_t val = (int32_t)cpu_ldl_data(env, rs1 + i * 4);
        cpu_stl_data(env, rd + i * 4, (uint32_t)(val > 0 ? val : 0));
    }
}
```

Key points:
- `cpu_ldl_data()` reads a 32-bit little-endian value (INT32).
- `cpu_stl_data()` writes a 32-bit little-endian value.
- For in-place operation (rd == rs1), each element is read before being written, so no special handling is needed — the sequential access pattern ensures correctness.
- No GPR writeback needed.

### 4. Translation Function: `target/riscv/insn_trans/trans_xg233ai.c.inc`

Add `trans_vrelu()` after `trans_vdot()` (after line ~70). Follows the same pattern as `crush`/`expand` — all operands are `get_gpr()` since this is a memory-only operation:

```c
static bool trans_vrelu(DisasContext *ctx, arg_r *a)
{
    if (!ctx->cfg_ptr->ext_xg233ai) {
        return false;
    }

    TCGv rd = get_gpr(ctx, a->rd, EXT_NONE);
    TCGv rs1 = get_gpr(ctx, a->rs1, EXT_NONE);
    TCGv rs2 = get_gpr(ctx, a->rs2, EXT_NONE);

    gen_helper_vrelu(tcg_env, rd, rs1, rs2);
    return true;
}
```

Key points:
- Uses `get_gpr()` for all three operands (no GPR writeback).
- `gen_helper_vrelu(tcg_env, rd, rs1, rs2)` — all memory operations, no return TCGv.
