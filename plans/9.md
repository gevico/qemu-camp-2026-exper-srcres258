# Plan: 为 Xg233ai `gemm` 指令实现 TCG 指令翻译

## 背景

`gemm` 是 Xg233ai 自定义扩展中第 9 条指令（§3.12），实现 INT32 4×4 矩阵乘法 C = A × B。编码为 R-type，opcode=`0x7b`，funct3=`6`，funct7=`0001110`（十进制 14）。

当前仓库中，Xg233ai 的 10 条指令已有 8 条实现了完整的 Decodetree 译码 → trans_ 翻译 → Helper 实现链路。`gemm` 和 `vadd` 尚未实现。测试文件 `test-insn-gemm.c` 已存在并已在 `Makefile.softmmu-target` 的 `TEST_CASES` 中注册。

## 实现方案

`gemm` 指令属于"数组到数组的操作"类别，不回写通用寄存器（`rd` 编码目标内存地址），因此使用与 `dma`、`vrelu`、`vscale` 等相同的 void helper 模式。

需要修改 **3 个文件**，按顺序：

### 步骤 1：在 decodetree 中添加 `gemm` 译码条目

**文件**：`target/riscv/xg233ai.decode`

在现有 `dma` 条目（funct7=`0000110`）之后，添加 `gemm` 条目（funct7=`0001110`）：

```
gemm     0001110  ..... ..... 110 ..... 1111011  @r
```

这会让 decodetree 工具自动生成 `decode_xg233ai()` 函数中对 `gemm` 的译码支持，并产生对 `trans_gemm` 的调用。

### 步骤 2：在 `helper.h` 中声明 helper 函数

**文件**：`target/riscv/helper.h`（第 1296 行之后）

`gemm` 不回写 GPR，使用 void 返回的 4 参数 helper：

```c
DEF_HELPER_FLAGS_4(gemm, TCG_CALL_NO_WG, void, env, tl, tl, tl)
```

参数顺序为 `(env, rd, rs1, rs2)`，对应 `(env, C地址, A地址, B地址)`，与现有 void helper 模式一致。

### 步骤 3：在 `trans_xg233ai.c.inc` 中添加翻译函数

**文件**：`target/riscv/insn_trans/trans_xg233ai.c.inc`

遵循 void helper 模式（同 `vrelu`、`vscale` 的结构）：

```c
static bool trans_gemm(DisasContext *ctx, arg_r *a)
{
    if (!ctx->cfg_ptr->ext_xg233ai) {
        return false;
    }

    TCGv rd = get_gpr(ctx, a->rd, EXT_NONE);
    TCGv rs1 = get_gpr(ctx, a->rs1, EXT_NONE);
    TCGv rs2 = get_gpr(ctx, a->rs2, EXT_NONE);

    gen_helper_gemm(tcg_env, rd, rs1, rs2);
    return true;
}
```

- `rd` = 矩阵 C 的内存地址（目标），使用 `get_gpr` 读取（因为是输入地址，不是写入目标寄存器）
- `rs1` = 矩阵 A 的内存地址
- `rs2` = 矩阵 B 的内存地址

### 步骤 4：在 `op_helper.c` 中实现 helper 函数

**文件**：`target/riscv/op_helper.c`（第 891 行之后，`HELPER(vmax)` 之后）

根据 §3.12 伪代码，实现三重循环矩阵乘法：

```c
void HELPER(gemm)(CPURISCVState *env, target_ulong rd,
                  target_ulong rs1, target_ulong rs2)
{
    for (int i = 0; i < 4; i++) {
        for (int j = 0; j < 4; j++) {
            int64_t acc = 0;
            for (int k = 0; k < 4; k++) {
                int32_t a = (int32_t)cpu_ldl_data(env, rs1 + (i * 4 + k) * 4);
                int32_t b = (int32_t)cpu_ldl_data(env, rs2 + (k * 4 + j) * 4);
                acc += (int64_t)a * (int64_t)b;
            }
            cpu_stl_data(env, rd + (i * 4 + j) * 4, (uint32_t)(int32_t)acc);
        }
    }
}
```

关键点：
- 使用 `cpu_ldl_data` 读取 INT32 元素，显式转型为 `int32_t`
- 累加器使用 `int64_t`（64 位精度）
- 行优先存储：`A[i][k]` = `rs1 + (i*4 + k)*4`，`B[k][j]` = `rs2 + (k*4 + j)*4`
- 使用 `cpu_stl_data` 写回结果，截断为 INT32

### 步骤 5：构建与测试验证

无需修改构建系统——`test-insn-gemm` 已在 `TEST_CASES` 中注册，构建系统会自动编译并运行。

```bash
nix develop
make -f Makefile.camp configure   # 重新配置以重新生成 decodetree
make -f Makefile.camp build       # 构建
make -C build check-gevico-tcg    # 运行 TCG 测试
```

预期输出中 `TEST 9/10 test-insn-gemm on riscv64` 不应有报错信息。
