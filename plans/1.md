# 实现 Xg233ai `dma` 指令计划

## 背景

测试文件 `tests/gevico/tcg/riscv64/test-insn-dma.c` **已经存在且完整**，包含三个测试用例（8×8 / 16×16 / 32×32）。缺失的是 QEMU TCG 后端的指令模拟——Xg233ai 扩展尚未在 `target/riscv/` 中注册、解码或实现。

当前 CPU 型号 `TYPE_RISCV_CPU_GEVICO_CV1`（`target/riscv/cpu.c:3335`）被 G233 机器使用，但没有开启 `ext_xg233ai`。

## 指令编码

```
+---------+------+------+------+------+-------+
| 0000110 | rs2  | rs1  | 110  |  rd  |1111011|  dma
+---------+------+------+------+------+-------+
  funct7    rs2    rs1  funct3   rd    opcode
```

- opcode = `0x7b`（custom-3）
- funct3 = `6`（`110`）
- funct7 = `6`（`0000110`）
- 内联汇编：`.insn r 0x7b, 6, 6, rd, rs1, rs2`

## 实现步骤

### 步骤 1 — 在 CPU 配置中添加扩展布尔字段

**文件**：`target/riscv/cpu_cfg_fields.h.inc`

在"Vendor-specific custom extensions"区块末尾（`ext_xmipslsp` 之后，约第 155 行）添加：

```c
BOOL_FIELD(ext_xg233ai)
```

### 步骤 2 — 添加扩展谓词函数

**文件**：`target/riscv/cpu_cfg.h`

在 `MATERIALISE_EXT_PREDICATE(XVentanaCondOps)` 之后（约第 71 行）添加：

```c
MATERIALISE_EXT_PREDICATE(xg233ai)
```

这会生成 `has_xg233ai_p()` 函数，供 decoder table 的 guard 使用。

### 步骤 3 — 创建 decodetree 描述文件

**新文件**：`target/riscv/xg233ai.decode`

```
#
# Xg233ai custom instruction decode
#

%rs2  20:5
%rs1  15:5
%rd    7:5

&r    rd rs1 rs2  !extern

@r    .......  ..... ..... ... ..... .......  &r  %rs2 %rs1 %rd

dma      0000110  ..... ..... 110 ..... 1111011  @r
```

参考 `target/riscv/XVentanaCondOps.decode` 的格式。目前仅解码 `dma` 一条指令（funct7=6），后续可扩展其他 Xg233ai 指令。

### 步骤 4 — 声明 helper 函数

**文件**：`target/riscv/helper.h`

在文件末尾添加：

```c
/* Xg233ai */
DEF_HELPER_FLAGS_4(dma, TCG_CALL_NO_WG, void, env, tl, tl, tl)
```

参数：env, dst（rd 寄存器值）, src（rs1 寄存器值）, grain（rs2 寄存器值）。

### 步骤 5 — 实现 helper 函数

**文件**：`target/riscv/op_helper.c`

在文件末尾添加 helper 实现：

```c
void HELPER(dma)(CPURISCVState *env, target_ulong dst,
                 target_ulong src, target_ulong grain)
{
    const int sizes[] = {8, 16, 32};
    int n = sizes[grain];

    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            uint32_t val = cpu_ldl_data(env, src + (i * n + j) * 4);
            cpu_stl_data(env, dst + (j * n + i) * 4, val);
        }
    }
}
```

使用 `cpu_ldl_data()` / `cpu_stl_data()` 进行 guest 内存访问（32 位 load/store，对应 FP32 的 4 字节宽度）。通过逐元素 copy 实现矩阵转置：`dst[j*N+i] = src[i*N+j]`。

### 步骤 6 — 创建 TCG 翻译函数

**新文件**：`target/riscv/insn_trans/trans_xg233ai.c.inc`

```c
static bool trans_dma(DisasContext *ctx, arg_r *a)
{
    if (!ctx->cfg_ptr->ext_xg233ai) {
        return false;
    }

    TCGv rd = tcg_constant_tl(a->rd);
    TCGv rs1 = tcg_constant_tl(a->rs1);
    TCGv rs2 = tcg_constant_tl(a->rs2);

    gen_helper_dma(cpu_env, rd, rs1, rs2);
    return true;
}
```

注意：这里 `a->rd`、`a->rs1`、`a->rs2` 是寄存器编号。但 `dma` 指令语义是读取这些寄存器的值作为地址/参数，因此需要通过 `get_gpr()` 获取寄存器值而非寄存器编号。

**修正**：应使用 `get_gpr()` 获取寄存器值：

```c
static bool trans_dma(DisasContext *ctx, arg_r *a)
{
    if (!ctx->cfg_ptr->ext_xg233ai) {
        return false;
    }

    TCGv dst = get_gpr(ctx, a->rd, ExtOp::EXT_NONE);
    TCGv src = get_gpr(ctx, a->rs1, ExtOp::EXT_NONE);
    TCGv grain = get_gpr(ctx, a->rs2, ExtOp::EXT_NONE);

    gen_helper_dma(cpu_env, dst, src, grain);
    return true;
}
```

## 风险与注意事项

1. **寄存器值 vs 寄存器编号**：`dma` 指令使用 rs1/rs2/rd 的值（地址和粒度参数），trans 函数中必须用 `get_gpr()` 取值，不能直接传寄存器编号。
2. **内存访问安全**：helper 中使用 `cpu_ldl_data()` / `cpu_stl_data()` 会自动处理 TLB 和权限检查，如果访问非法地址会产生异常。
3. **grain 越界**：规格说明 grain 超出 `{0,1,2}` 时行为未定义，helper 中不做额外检查。
4. **区域重叠**：规格说明 src 和 dst 不得重叠，helper 中不做检查。
5. **funct7 编码冲突**：XVentanaCondOps 使用 funct7=0 的 custom-3 空间，xg233ai 使用 funct7=6，不冲突。但需确认 decoder table 的优先级顺序——如果 XVentanaCondOps 先匹配但 funct7 不同则不会误匹配（decodetree 只匹配完整模式）。
